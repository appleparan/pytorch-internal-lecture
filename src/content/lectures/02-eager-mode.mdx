# 왜 이렇게 복잡한거지? (1/2)

**`torch.matmul` 자체가 생각보다 복잡함**

- Input tensor의 shape에 따라 상당히 많은 경우의 수가 발생
  - 1D vector * 1D vector인 경우
  - 2D matrix * 1D vector인 경우
  - 2D matrix * 2D matrix인 경우
  - 3D 이상의 tensor가 input으로 들어온 경우
    - 두 tensor의 dimension이 일치하지 않아 broadcast가 필요한 경우

**Language boundary를 여러차례 넘어야 할 수도 있음**

- Python에서 C++로 전환
- GPU를 쓰기 위해 다시 C++에서 CUDA로 전환

</Slide>

<Slide>

# 왜 이렇게 복잡한거지? (2/2)

**꽤 다양한 실행 시나리오를 지원해야 함**

- 계산이 CPU에서 이루어질 수도 있고
  - GPU에서 이루어 질 수도 있고
  - 또 다른 종류의 device에서 이루어질 수도 있음
- Back propagation을 위한 dynamic computational graph가 생성되어야 할 수도 있고
  - 그렇지 않을수도 있고
- Trace 과정일 수도 있고
  - 아닐 수도 있고

# Python에서 C++ code로 넘어가는 과정 (1/2)

**Pytorch는 자체 custom binding 기술 사용**

- 성능과 확장성 요구사항을 충족하기 위해

# [참고] native_functions.yaml

`native_functions.yaml`은 ATen library에서 **operator를 정의하고 등록하는 파일**

```yaml {1, 2, 3, 4-8, 9}
- func: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  device_check: NoCheck         # 디바이스 검사를 비활성화
  structured_delegate: add.out  # add.out의 정의를 상속
  variants: function, method    # namespace 함수 + Tensor 메서드
  dispatch:
    SparseCPU, SparseCUDA, SparseMeta: add_sparse
    SparseCsrCPU, SparseCsrCUDA, SparseCsrMeta: add_sparse_csr
    MkldnnCPU: mkldnn_add
    ZeroTensor: add_zerotensor
    NestedTensorCPU, NestedTensorCUDA: NestedTensor_add_Tensor
  tags: [core, pointwise]
```

- `func`: 함수 signature (이름, 인자, 반환 타입)
- `variants`: `function` (namespace 호출) / `method` (Tensor 메서드 호출)
- `dispatch`: dispatch key 또는 backend별 impl 함수 지정

</Slide>

# Tensor의 구조

<Mermaid>
{`graph LR
    T["at::Tensor"] --> TI["c10::TensorImpl"]
    TI --> S["c10::Storage"]
    S -->|shared ptr| SI["c10::StorageImpl"]
    TI --> MD["metadata"]
    TI --> AG["autograd"]`}
</Mermaid>

- Python `torch.tensor`는 C++ `at::Tensor`에 대응
- `c10::TensorImpl`이 실제 구현체
  - `at::Tensor`가 `c10::TensorImpl`을 member로 가지고 있음
- `c10::Storage`에 data가 보관
  - 여러 `at::Tensor`가 같은 `c10::StorageImpl`을 공유 가능
  - `c10::Storage`는 reference count pointer로 관리됨
- **metadata**: storage offset, size, stride, elem 갯수, dtype
- **autograd**: gradient, grad graph, variable version

# Tensor의 Lifecycle: 소멸

**`at::Tensor`의 소멸**
- Python에서 `torch.tensor`가 제거되는 시점에 소멸
  - 명시적으로 객체를 제거 (`del` keyword 사용)
  - 또는 garbage collection에 의해 객체가 제거될 때
- `THPVariable_subclass_dealloc`이 entry point

**`c10::Storage`의 소멸**
- `at::Tensor`에서 reference count ptr로 관리
- `c10::Storage`를 공유하는 `at::Tensor`들이 **모두 소멸돼야** 할당된 memory가 정리됨

# Stream: 개념

**Heterogeneous computing에서의 실행 모델:**

- Host가 device로 command, data 등을 전달 → device는 이를 처리
- Host는 device 완료를 기다리지 않고 **asynchronous**하게 동작

**Stream**: host가 device로 전달한 command가 담긴 **queue**

- Host는 stream에 command를 삽입
- Device는 stream에서 **FIFO 순서**로 command를 처리
- 각 device마다 **stream pool**이 존재

<div class="text-sm text-gray-400 mt-2">

# Event: 개념 (2/2)

**`event.record(stream)`**

- Stream에 record command를 삽입

# Event: 구현 방식

# Event: API Details (1/3)

**`record(stream)`**: stream에 record command를 삽입

- Stream과 event의 device type, index는 같아야 함
- 호출 경로: `c10::Event::record` → `InlineEvent::record` → `VirtualGuardImpl::record` → device-specific impl
  - **XPU**: `sycl::queue`에 `ext_oneapi_submit_barrier` 호출
  - **CUDA**: `cudaEventRecord` 호출
- 수행 후 `was_marked_for_recording_`을 true로 변경

**`recordOnce(stream)`**: record가 한 번만 호출되도록 보장

- Event는 thread-safe하지 않아서 사용
- `was_marked_for_recording_`이 false일 때만 record 실행

</Slide>

<Slide>

# Event: API Details (2/3)

**`block(stream)`**: record command가 처리될 때까지 대기하라는 block command를 삽입

- Stream과 event의 **device type**은 같아야 하지만 **device index는 다를 수 있음**
- 같은 device index여도 deadlock이 발생하지 않음
  - Block command가 처리되기 전까지 계속 command를 처리해 record command를 처리할 수 있으므로
- 호출 경로: `c10::Event::block` → `InlineEvent::block` → `VirtualGuardImpl::block` → device-specific impl
  - **XPU**: `sycl::queue`에 `ext_oneapi_submit_barrier` 호출 (sycl::event list를 argument로)
  - **CUDA**: `cudaStreamWaitEvent` 호출

</Slide>

<Slide>

# Event: API Details (3/3)

**`query()`**: record command가 처리되었는지 여부를 반환

- Record를 호출한 적이 없으면 항상 true를 반환
- **XPU**: `sycl::event`의 `command_execution_status`가 `complete`인지 확인
- **CUDA**: `cudaEventQuery` 호출

**`synchronize()`**: host가 record command가 처리될 때까지 대기

- **XPU**: `sycl::event`에 `wait_and_throw` 호출
- **CUDA**: `cudaEventSynchronize` 호출

</Slide>

# DeviceGuard: 개념

**DeviceGuard**: 특정한 device, stream, event의 안전한 사용을 보장하는 장치

- 정해진 scope에서 특정 device를 set하고, scope을 벗어나면 원래 device로 reset
- **RAII** (Resource Acquisition Is Initialization) 디자인 패턴
- Nesting된 형태로 사용 가능

```
┌─────────────────────────────────┐
│  device1                        │
│  ┌───────────────────────────┐  │
│  │  DeviceGuard (1 → 2)     │  │
│  │  ┌─────────────────────┐  │  │
│  │  │  DeviceGuard (2 → 3)│  │  │
│  │  │  device3             │  │  │
│  │  └─────────────────────┘  │  │
│  │  device2 (restored)       │  │
│  └───────────────────────────┘  │
│  device1 (restored)             │
└─────────────────────────────────┘
```

# Directory Structure: aten

<div class="text-sm">

`aten/src/ATen/`

- **`core/`**: Tensor 관련 핵심 선언과 구현
  - `boxing/`, `dispatch/`, `op_registration/`: operator dispatch 관련
- **`cuda/`, `hip/`, `metal/`, `mps/`, `vulkan/`, `xpu/`**: device dependent 구현 (runtime 등)
  - CUDAEvent, XPUDevice 등
  - 일부 directory에는 device 전용 계산 library 구현도 포함
- **`native/`**: operator 구현체들, **`native_functions.yaml`** 포함
  - Device independent한 구현체는 바로 밑에
  - Device dependent한 구현체는 sub directory (`cpu/`, `cuda/`, `cudnn/`, `hip/`)
- **`templates/`**: gen source code들의 뼈대
- **`test/`**: ATen test code

</div>

</Slide>

<Slide>

# Directory Structure: c10

<div class="text-sm">

`c10/`

- **`core/`**: PyTorch 핵심 구현과 선언, aten이 상속받아 구현
  - Device 관련: `Allocator`, `Device`, `DeviceGuard`, `Event`, `Stream` 등
  - Dispatch 관련: `DispatchKey`, `DispatchKeyset` 등
  - Tensor 관련: `TensorImpl`, `StorageImpl` 등
- **`cuda/`, `xpu/`, `hip/`, `mobile/`**: 특정 device의 전용 구현
  - `CUDACachingAllocator`, `XPUStream` 등
- **`macros/`**: compiler에서 해석할 inline attributes 관련 macro
  - Shared library visibility 조절, warning 강제 무시 등
- **`test/`**: C10 test code
- **`util/`**: helper class/method들
  - Environments, exception, backtrace 등
  - Array, BFloat16, Complex 등

</div>

</Slide>
